### Methods dropped from Phase 1

Two methods were cut from Phase 1 scope due to `FacesContext.getCurrentInstance()` calls that occur before any mockable guard, making them unreachable without a JSF runtime or a FacesContext test helper:

- **`com.dotachile.torneos.service.TorneoService.inscribirClanTorneo`** — reads servlet init-params (`com.tarreo.dota.torneo.minPlayersPorClan`, `maxPlayersPorClan`) via `FacesContext` at approximately line 1301, before any business-logic branch. Two tests were intended: `inscribirClanTorneoRejectsWhenPhaseIsNotRegistration` and `inscribirClanTorneoRejectsDuplicateClan`. Both were dropped. Note: `inscribirClanTorneoForced` (adjacent method, out of Phase 1 scope) skips the FacesContext read entirely and IS testable — Phase 2 should cover it as a direct replacement for the dropped tests.

- **`com.dotachile.clanes.service.ClanService.aceptarInvitacion`** — calls `FacesContext.getCurrentInstance()` at approximately line 279, immediately after the ban check and before any mockable roster logic, leaving no reachable branch. One intended test was dropped: `aceptarInvitacionAddsNewMemberToPeones`.

### Tooling issue discovered

**Mockito 3 / ByteBuddy `ClassFormatError` on `ConfirmacionFacade`**

`ConfirmacionFacade.findByTag()` catches `javax.persistence.NoResultException` from a JPA API stub jar (the jar contains no method bodies — only signatures). When Mockito 3 / ByteBuddy attempts to generate a mock subclass, it encounters the absent `Code` attribute and throws `ClassFormatError: Absent Code attribute`, aborting the test class.

Workaround used in `ClanServiceTest`: simply do not mock or inject `ConfirmacionFacade` at all, since none of the in-scope `ClanService` methods under test actually invoke it.

This workaround is not universally available. If Phase 2 target methods do call `ConfirmacionFacade`, the team must adopt one of the following before writing those tests:

1. Replace the stub javax.persistence API jar with a real one that includes method bodies (preferred — fixes the root cause for all affected facades).
2. Add a thin interface or delegate seam over `ConfirmacionFacade` so Mockito mocks the seam, not the concrete class.
3. Audit every other facade in the codebase for the same pattern (`catches JPA exception from stub jar`) and pre-emptively apply option 1 or 2 before reaching those facades.

## Phase 2 goals

1. **Push in-scope service-layer coverage to a per-package floor of 60% line / 40% branch.** The current baseline above shows exactly how far each package must move: `com.dotachile.torneos.service` needs the largest jump (from 12.9% line), while `com.dotachile.shared` already exceeds the line floor but still needs branch work.

2. **Bring `BasicService` from 0% to a meaningful baseline.** `com.dotachile.torneos.service.BasicService` is the single biggest gap inside the in-scope packages — it is approximately 563 LoC and contributes heavily to the low torneos-service numbers. Getting it to the 60%/40% floor will dominate the torneos-service delta.

3. **Cover `EloSystem` branches.** The lookup table has 50+ rating-difference bands; Phase 1 reached 98.3% line / 37.7% branch. The remaining branch gap is entirely in the band-edge conditionals. Phase 2 should add boundary tests at key band transitions — for example: dif=3 vs dif=4, dif=10 vs dif=11, and the high end around dif=735 vs dif=736. Target: approach 100%/100% for this class.

4. **Bring ManagedBeans into the JaCoCo report** by removing the `**/*MB.class` exclusion AND introducing a `FacesContext` test helper (static mock, test-scoped `FacesContext` stub, or equivalent) that allows JSF-aware code to run in plain JUnit. Use the same helper to cover `Util`'s JSF helpers (`addErrorMessage`, `addInfoMessage`, `addFatalMessage`, `addWarnMessage`, `keepMessages`, `navigate`, `getUsuarioLogeado`). Once the helper exists, cover MBs class-by-class in priority order.

5. **Test the dropped Phase 1 methods.** Either via the FacesContext helper (covers `inscribirClanTorneo` and `aceptarInvitacion` end-to-end) or via `*Forced` variants where they exist (`inscribirClanTorneoForced` bypasses the FacesContext read and should be covered regardless). Both original dropped tests must appear in the Phase 2 test suite in one form or the other.

## Coverage target policy

Phase 2 sets a **per-package floor**, not a global percentage. Target: **60% line and 40% branch** on every package in the "TEST" or "TEST further" rows above. Packages in "EXCLUDE" rows are excluded from the floor.

The aim is to make new uncovered code visible at PR time — not to chase 100% on legacy code that doesn't justify the effort. Any package that already exceeds the floor at the end of Phase 1 (e.g. `com.dotachile.shared` at 85% branch) must not regress below it.

## Doc bug noted (out of scope for Phase 2)

`com/dotachile/clanes/CLAUDE.md` glossary implies `Movimiento` lives in `com.dotachile.clanes.entity`. The actual package is `com.dotachile.media`. This is a one-line fix that belongs in a separate docs commit, not in Phase 2 test work.

## Out of scope for Phase 2

- Literal 100% coverage anywhere.
- Container or integration tests with a real Payara or PostgreSQL instance.
- PowerMock under any circumstances.
- Refactoring production code beyond the minimum needed to make a target testable. The exception is the explicit "extract business logic from MBs into testable collaborators" goal, which is an authorized refactor with a clear scope: only MBs that cannot be covered via the FacesContext helper alone.

## Definition of done (Phase 2)

- Every package marked TEST or TEST further meets the 60% line / 40% branch floor in the JaCoCo report generated by `mvn -o verify`.
- Phase 1's two dropped methods (`inscribirClanTorneo`, `aceptarInvitacion`) are tested — either via the FacesContext helper or, where applicable, via their `*Forced` variants (`inscribirClanTorneoForced` for the first).
- The Mockito/ByteBuddy `ClassFormatError` issue is either resolved (real javax.persistence API jar with method bodies) or worked around via documented seams, and the resolution is recorded in a follow-up commit or ADR.
- A new JaCoCo HTML report is generated at the end of Phase 2 and the per-package numbers are recorded here or in a successor document.
- A Phase 3 decision is recorded: either "stable test posture — no further phases planned" or a scoped Phase 3 spec covering remaining gaps.
